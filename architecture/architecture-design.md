# 🎯 路由系统完整架构设计

> 版本：v1.0
> 撰写日期：2026-06-01
> 背景：解决子程序分配混乱、同一话题被拆散、状态管理缺失的问题
> 基于：OpenClaw 主会话(MFDoom) + Route Server + 三方消息通道

---

## 一、核心问题与设计目标

### 现状痛点

| # | 问题 | 后果 |
|:-:|:-----|:-----|
| 1 | **同一话题拆散**（如"语音识别"）被拆成多个子程序并行跑 | 资源浪费、结果碎片化 |
| 2 | **没有统一状态表** | 不知道谁在跑、跑完了没、结果在哪 |
| 3 | **子程序自生自灭** | 崩溃后无人接管，老板以为在跑实际已挂 |
| 4 | **决策无通道** | 子程序需要老板选方案时没地方传达 |
| 5 | **无进度可视化** | 老板想看进度只能干等 |

### 设计目标

```
✅ 统一注册 → 所有子程序都在 Route Server 注册，一个话题一个任务
✅ 话题聚合 → 新消息优先匹配已有同类任务，合并不拆散
✅ 可视监控 → GUI 实时显示每个任务的状态/进度/结果
✅ 决策通道 → 子程序需决策 → GUI弹出 + 聊天推送 → 老板任选通道回复
✅ 崩溃恢复 → 挂掉重启后恢复队列，不丢任务
✅ 零依赖    → 只用 Node.js 内置模块，不引入 npm
```

### 组件职责划分

| 组件 | 语言 | 做什么 | 不做什么 |
|:-----|:----|:-------|:---------|
| **主会话 (MFDoom)** | OpenClaw Agent | 接消息→POST任务→轮询→创建子程序→汇总回复 | 不做路由判断、不做实际计算 |
| **Route Server** | Node.js (zero-dep) | 队列调度、话题聚合、状态管理、决策分发、Worker管理 | 不处理业务逻辑、不分析数据 |
| **AI子程序** | OpenClaw SubAgent | 具体分析/推理/生成 | 不管理状态、不调度任务 |
| **本地Worker** | Node.js 子进程 | 文件操作/脚本执行/系统命令 | 不做AI推理 |
| **桌面GUI** | 浏览器 SPA | 实时展示任务状态、决策交互 | 不做后端逻辑 |

## 三、核心机制详解

### 3.1 话题聚合 — 解决"同一话题被拆散"

这是当前最关键的改进。以前：每条消息进来都创建新子程序。
现在：先查 Route Server 是否有同类话题正在运行。

**匹配流程：**

```
消息进来 "语音识别用百度"
        │
        ▼
Route Server 查话题注册表 (topic-registry)
        │
        ├── 话题 "语音识别" 已有任务在运行？
        │      ├── ✅ 是 → 追加到现有任务（消息合并），不新建
        │      │              ↓
        │      │        更新现有任务的 context（增加来源消息）
        │      │        返回 sessionKey（复用现有子程序）
        │      │
        │      └── ❌ 否 → 创建新任务
        │                    ↓
        │              注册话题 "语音识别"
        │              创建子程序 → 开始处理
        │
        ├── 后续同类消息 → 自动路由到同一子程序
        └── 子程序完成 → 从话题注册表移除
```

**话题合并规则：**

| 场景 | 行为 |
|:-----|:-----|
| 完全相同的话题 | 合并到现有任务，追加材料 |
| 话题有包含关系（如"语音识别"和"语音识别用百度"） | 合并到父话题任务 |
| 话题不同但相关（如"语音识别"和"百度ASR"） | 按关键词相似度打分，>阈值合并 |
| 完全不相关 | 创建新任务 |

**话题注册表数据结构：**

```json
{
  "topics": {
    "语音识别": {
      "activeTaskId": "task-abc-123",
      "sessionKey": "subagent_xxxx",
      "createdAt": "2026-06-01T14:30:00Z",
      "mergedMessages": [
        { "text": "语音识别用百度", "channel": "dingtalk" },
        { "text": "语音识别顺便把这段也翻译了", "channel": "dingtalk" }
      ]
    }
  }
}
```

### 3.2 状态管理 — 每个子程序的完整生命周期

**状态机：**

```
          ┌──────────┐
          │  queued   │ ← 新任务入队
          └────┬─────┘
               │ 调度器出队
               ▼
          ┌──────────┐
          │  assigned │ ← 已分配 Worker/子程序
          └────┬─────┘
               │ Worker 确认接收
               ▼
          ┌──────────┐
          │  running  │ ← 正在处理
          └────┬─────┘
               │
     ┌─────────┼─────────┐
     │         │         │
     ▼         ▼         ▼
  ┌────────┐┌──────────────┐┌────────┐
  │awaiting││  completed   ││ failed │
  │decision││              ││        │
  └───┬────┘└──────────────┘└────────┘
      │ 老板回复决策
      ▼
  ┌──────────┐
  │  running  │ ← 继续处理
  └────┬─────┘
       │
       ▼
  ┌──────────┐
  │completed │
  └──────────┘
```

**持久化（崩溃恢复）：**

```javascript
// data/task-registry.json（每 30 秒自动快照）
{
  "tasks": {
    "task-abc-123": {
      "id": "task-abc-123",
      "name": "语音识别",
      "message": "语音识别用百度",
      "source": { "channel": "dingtalk", "user": "老板" },
      "status": "running",        // queued | assigned | running | awaiting_decision | completed | failed
      "topic": "语音识别",
      "priority": 7,
      "progress": 65,             // 0-100
      "subagentInfo": {
        "sessionKey": "subagent_xxxx",
        "model": "deepseek/deepseek-v4-flash",
        "startedAt": "2026-06-01T14:30:00Z",
        "completedAt": null
      },
      "decisionRequest": null,    // 或 { prompt, options, responded }
      "result": null,
      "errors": [],
      "workerPid": null,
      "createdAt": "2026-06-01T14:30:00Z",
      "updatedAt": "2026-06-01T14:32:00Z",
      "timeoutMs": 300000
    }
  },
  "lastSave": "2026-06-01T14:32:00Z",
  "version": 1
}
```

**崩溃恢复流程：**

```
Route Server 启动
      │
      ├── 检查 data/task-registry.json
      │
      ├── 存在 → 恢复任务列表
      │         ├── status=queued → 保持（等待重新调度）
      │         ├── status=running → 重置为 queued（Worker已死）
      │         ├── status=awaiting_decision → 保持（等待老板回复）
      │         │    └── decisions/pending/ 中有对应文件
      │         ├── status=completed → 归档忽略
      │         └── status=failed → 通知主会话（如需）
      │
      ├── 清理 topic-registry（所有 active 任务变成 zombie → 移除）
      └── 恢复 WebSocket → GUI 主动重连
```

### 3.3 决策流程 — 子程序需要老板选方案时

完整链路：

```
子程序（AI 或 Worker）需要老板决策
      │
      ├── POST /api/decisions { taskId, prompt, options }
      │     示例：
      │     {
      │       "taskId": "task-abc-123",
      │       "prompt": "语音识别服务用哪个方案？",
      │       "options": [
      │         { "key": "A", "label": "百度语音(免费,95%准确率)" },
      │         { "key": "B", "label": "阿里云语音(￥0.01/次,98%准确率)" }
      │       ]
      │     }
      │
      ├── Route Server:
      │     ① 任务状态 → awaiting_decision
      │     ② 写 decisions/pending/{taskId}.json
      │     ③ WebSocket → GUI 推送 decision:new
      │     ④ GUI 出现 [待决策] 卡片 + 选项按钮
      │
      ├── 主会话轮询 decisions/pending/（每2秒）
      │     ① 发现新文件
      │     ② 通过聊天通道推送：
      │        "🤔 需要您抉择：
      │         任务：语音识别用百度
      │         问题：语音识别服务用哪个方案？
      │         A) 百度语音(免费,95%准确率)
      │         B) 阿里云语音(￥0.01/次,98%准确率)
      │         回复「选A」或「选B」"
      │
      ├── 老板回复（任选一）：
      │     ● 聊天回复 "选A"
      │     ● GUI 点击 [选A] 按钮
      │     ↓
      │     主会话/GUI → POST /api/decisions/{id}/respond { selection: "A" }
      │
      └── Route Server:
            ① 任务状态 → running
            ② 删除 pending 文件，归档到 resolved
            ③ WebSocket → decision:resolved
            ④ 返回子程序 { selected: "A" }
            ⑤ 子程序继续执行
```

**多决策并发消除歧义：**

老板回复 | 匹配规则
---------|----------
"选A" | 取最新的 pending 决策
"语音识别选A" | 匹配 prompt 或 task 名称含"语音识别"
"task-abc-123 选A" | 精确匹配 taskId
"选B，第二个方案" | 匹配选项 label 含"第二个方案"

### 3.4 优先级排序机制

**优先级计算公式：**

```
basePriority = 5（默认）

+3  老板直接 @ 或提到紧急词（"立刻/马上/紧急/坏了")
+2  deadlines / 截止时间在 1 小时内
+1  用户已等待超过 5 分钟（每次 +1，上限 +3）
 0  普通任务
-2  知识沉淀、离线分析、定时任务
-3  批量处理、低优先级维护

最终优先级 = clamp(basePriority + modifiers, 0, 10)
```

**队列调度策略：**

- 按优先级降序出队
- 同优先级按 FIFO
- 高优先级任务可以插队（不阻塞正在运行的，但下一个出队的一定是最高优先级）
- 最大并发数：受限 GTX 960 4GB → AI 子程序最多 **3 个并发**，本地 Worker 最多 **5 个并发**

### 3.5 超时与异常处理

```
每个任务有超时时间：
  ├── 本地 Worker：60 秒
  ├── AI 子程序：300 秒（5分钟）
  └── 决策等待：无限制（但老板不回复则每 5 分钟提醒一次）

超时处理：
  ① 任务标记为 failed, reason="timeout"
  ② kill 对应 Worker（如果还活着）
  ③ 通知主会话 → 通知老板
  ④ 写入日志

Worker 异常退出：
  ① child_process.on('exit', code)
  ② code !== 0 → 任务标记 failed
  ③ 自动重试（最多 2 次）
  ④ 重试仍失败 → 通知主会话
```

## 四、Route Server API 规范

### 4.1 REST API

```http
### 健康检查
GET /ping
→ 200 { "status": "ok", "uptime": 12345, "queueLength": 3, "workers": 2 }

### 提交新任务
POST /api/tasks
Body: {
  "message": "语音识别用百度",
  "channel": "dingtalk",
  "user": "老板",
  "priority": 7
}
→ 201 { "taskId": "task-abc-123", "matched": true, "existingTask": null }
   // matched=true 表示匹配到已有话题，merged=true 表示合并到现有任务

### 获取待 AI 处理的任务（主会话轮询）
GET /api/tasks/pending-ai
→ 200 [{ "taskId": "task-abc-123", "name": "语音识别", "message": "..." }]

### 更新任务状态（主会话/Worker 调用）
POST /api/tasks/:id/update
Body: { "status": "running", "progress": 50 }
→ 200 { "ok": true }

### 任务完成回调
POST /api/tasks/:id/callback
Body: { "status": "completed", "result": { ... }, "durationMs": 23000 }
→ 200 { "ok": true }

### 提交决策请求
POST /api/decisions
Body: { "taskId": "...", "prompt": "...", "options": [...] }
→ 201 { "decisionId": "dec-xyz" }

### 响应决策
POST /api/decisions/:id/respond
Body: { "selection": "A", "source": "dingtalk" }  // source: gui|dingtalk|wechat|feishu
→ 200 { "ok": true }

### 获取任务列表（GUI 用）
GET /api/tasks?status=all
→ 200 { "tasks": [...], "stats": { "queued": 2, "running": 3, "completed": 47, "failed": 1 } }

### 获取任务详情
GET /api/tasks/:id
→ 200 { "id": "...", "name": "...", ... }
```

### 4.2 WebSocket 事件

| 事件 | 触发时机 | 负载 |
|:-----|:---------|:-----|
| `task:new` | 新任务入队 | `{ id, name, channel, priority }` |
| `task:status` | 状态/进度变化 | `{ id, status, progress? }` |
| `task:completed` | 任务完成 | `{ id, result, duration }` |
| `task:failed` | 任务失败 | `{ id, errors }` |
| `decision:new` | 新决策请求 | `{ taskId, prompt, options }` |
| `decision:resolved` | 决策已响应 | `{ taskId, selected, source }` |
| `server:heartbeat` | 每 30 秒 | `{ uptime, tasks }` |

### 5.3 自启动方案

```powershell
# 方案A（推荐）：Windows 计划任务开机启动
$action = New-ScheduledTaskAction -Execute "node.exe" `
  -Argument "C:\path\to\route-server.js"
$trigger = New-ScheduledTaskTrigger -AtStartup -RandomDelay 00:01:00
$settings = New-ScheduledTaskSettingsSet -AllowStartIfOnBatteries -DontStopIfGoingOnBatteries
Register-ScheduledTask -TaskName "TaskRouteServer" `
  -Action $action -Trigger $trigger -Settings $settings `
  -User "SYSTEM" -RunLevel Highest

# 方案B（简易）：Startup 文件夹快捷方式
# 创建 route-server.cmd:
#   @echo off
#   cd /d C:\Users\Administrator\.openclaw\workspace
#   node route-server.js
# 放到 shell:startup
```

**Gateway 重启联动：**
- Gateway 启动时 → 主会话自动检查 Route Server 是否存活
- 未存活 → 执行 `exec("start node route-server.js")`
- 存活 → 继续（Route Server 比 Gateway 存活时间长，不需每次重启）

## 六、主会话集成方案

### 6.1 启动时

主会话 (MFDoom) 启动时运行以下逻辑：

```
function setupRouteServer():
  ① exec("node route-server.js --check", { timeout: 3000 })
     → 有输出？已存活 → return
     → 无输出？启动它 → exec("start /B node route-server.js")
  
  ② 设置定时器（每 30 秒）
     GET /ping → 200? 正常
     GET /ping → 失败? 重启 Route Server
```

### 6.2 消息处理流程

```
收到消息 → POST /api/tasks { message, channel, user }
  ↓
Route Server 返回 { taskId, matched, existingTask }
  ↓
主会话判断：
  ├── matched && existingTask == null → 新任务，需要创建子程序
  │    ① 等待 GET /api/tasks/pending-ai 返回此 taskId
  │    ② sessions_spawn(task="消息内容+上下文", model="deepseek-v4-flash")
  │    ③ 子程序完成 → POST /api/tasks/{id}/update { status, result }
  │    ④ 回复消息通道
  │
  ├── matched && existingTask != null → 已有任务正在处理
  │    ① 追加消息到现有任务（不创建新子程序）
  │    ② 回复：已收到，这个和正在处理的"语音识别"合并了
  │
  └── matched == false → 无匹配话题
       ① 如果没有其他话题可合并，创建新话题
       ② 同上 AI 处理流程
```

### 6.3 决策轮询

```
setInterval(2000):  // 每 2 秒
  ① 读取 decisions/pending/ 目录
  ② 发现新文件（not exists in local cache）
     → 读取文件内容
     → 推送到钉钉/微信/飞书（根据任务来源通道）
     → 记录到 local cache（防止重复推送）
  ③ 文件被删除（已响应）
     → 更新 local cache
```

### 6.4 主会话代码变更

现有的主会话操作手册（AGENTS.md）中，"消息进来→创建子程序"的流程需要调整为：

```
旧流程：
  收到消息 → 立即 sessions_spawn 创建子程序

新流程：
  收到消息 → POST /api/tasks（Route Server 决定聚合/新建）
            → 等待 pending-ai 返回
            → 才 sessions_spawn
```

这个改动很小，只在 `AGENTS.md` 中增加一步路由查询，不改变主会话的"只分拣不干活"核心原则。

## 八、实施计划（分 4 阶段）

### 阶段一：Route Server 骨架（2 天）

**目标**：Route Server 能跑起来，能接收任务、调度、管理状态、持久化。

| 步骤 | 产出 | 验收标准 |
|:-----|:-----|:---------|
| 1.1 HTTP 服务 (`route-server.js`) | 端口 :3456，POST/GET 路由 | `curl localhost:3456/ping` 返回 `ok` |
| 1.2 优先队列 (`lib/task-queue.js`) | 按优先级 + FIFO 出队 | 高优先级任务先出 |
| 1.3 任务注册表 (`lib/task-registry.js`) | 内存 Map + JSON 快照 | 停止后重启恢复队列 |
| 1.4 调度器 (`lib/scheduler.js`) | 每秒检查队列，出队并分配 | Worker 接到任务 |
| 1.5 Worker 管理器 (`lib/worker-manager.js`) | fork 子进程，通信，超时 | 任务完成回调正常 |

**验证方法**：手动 `curl POST /api/tasks` 提交任务 → 检查调度器出队 → 检查任务状态变化。

### 阶段二：话题聚合 + AI 子程序桥接（1.5 天）

**目标**：同一话题合并、主会话轮询 pending-ai、子程序状态上报。

| 步骤 | 产出 | 验收标准 |
|:-----|:-----|:---------|
| 2.1 话题注册表 (`lib/topic-registry.js`) | 按关键词聚合 | 同类消息返回 `existingTask` |
| 2.2 关键词匹配集成 | 复用 `route-task.js` 到 Route Server | 匹配阈值可配置 |
| 2.3 pending-ai 接口 | `GET /api/tasks/pending-ai` | 主会话可获取待处理任务 |
| 2.4 回调更新接口 | `POST /api/tasks/:id/update` | 子程序完成可上报结果 |

**验证方法**：发两条"语音识别"消息 → 第一条创建新任务，第二条返回 `existingTask`。

### 阶段三：桌面 GUI + 实时更新（1.5 天）

**目标**：浏览器打开能看到任务状态，WebSocket 实时刷新。

| 步骤 | 产出 | 验收标准 |
|:-----|:-----|:---------|
| 3.1 静态文件服务 | Route Server 提供 `/dashboard` | 浏览器打开显示页面 |
| 3.2 WebSocket (`lib/websocket.js`) | 任务状态变更推送到浏览器 | 状态变化页面自动刷新 |
| 3.3 GUI 界面 (`public/dashboard.html`) | 完整的面板布局 | 显示队列/运行/完成/决策 |
| 3.4 决策按钮 | GUI 上点击 [选A] 可响应 | 决策响应后状态更新 |
| 3.5 自启动 + 自动打开浏览器 | 开机启动，自动开标签 | 重启后看得到面板 |

**验证方法**：提交任务 → 浏览器面板实时显示状态变化 → 点击决策按钮 → 状态更新。

### 阶段四：决策通道 + 聊天集成 + 异常处理（1.5 天）

**目标**：子程序可请求决策、老板可在任何通道回复、崩溃恢复。

| 步骤 | 产出 | 验收标准 |
|:-----|:-----|:---------|
| 4.1 决策管理器 (`lib/decision-manager.js`) | 文件协议 + 回调 | Worker/AI 可请求决策 |
| 4.2 主会话决策轮询 | 主会话读取 pending 文件并推送 | 聊天通道收到决策消息 |
| 4.3 聊天回复解析 | "选A" → 路由回传 | 回复后任务继续 |
| 4.4 超时 + 重试机制 | 任务超时自动失败 | 卡住的任务自动标记 |
| 4.5 崩溃恢复 | 重启后恢复队列 | 验证：kill Route Server → 重启 → 任务恢复 |

**验证方法**：完整端到端测试：消息→路由→AI子程序→请求决策→老板回复→继续→完成→通知。

## 二、各层级职责与模型分配

### 2.1 主程序 (Main) — 本地 MFDoom

| 维度 | 说明 |
|:-----|:-----|
| **模型** | `ollama/MFDoom/deepseek-r1-tool-calling:1.5b`（本地免费）|
| **窗口** | 32K（保持轻量，用完即清）|
| **做** | 纯聊天直接回；任务指令→转发一级；接收最终结果→回复老板 |
| **不做** | 不做任何计算、分析、搜索、判断 |

**判断流程：**

```
收到消息
  ↓
主程序自行做简单判断：
  ├── 纯问候/闲聊/情绪表达？
  │    例："好的"、"辛苦了"、"嗯嗯"、"哈哈"、"收到"、"哦"
  │    → 直接本地 MFDoom 生成礼貌回复
  │    → 回复后结束，不创建任何子程序
  │
  ├── 明显是非任务类反馈/确认？
  │    例："好的收到"、"可以"、"先这样"、"没问题"
  │    → 简单回复确认
  │    → 回复后结束
  │
  └── 其他（包含具体指令/问题/要求）
       → 转发给一级子程序
       → 主程序不自己分析

> 注意：主程序不做精确意图分析，遇模棱两可一律转发一级。宁可误发，不可漏发。
```

### 2.2 一级子程序 (Orchestrator) — DeepSeek V4 Flash

| 维度 | 说明 |
|:-----|:-----|
| **模型** | `deepseek/deepseek-v4-flash`（云端按量）|
| **窗口** | 128K |
| **创建方式** | 主程序 `sessions_spawn(context="isolated")` |
| **做** | ① 聊天vs任务判断 ② 关键词匹配路由 ③ 汇总结果 |
| **不做** | 不做具体计算、不写代码、不查文件 |

#### 判断机制：聊天 vs 任务

一级子程序收到消息后，执行以下判断流程：

```
收到消息（来自主程序转发的非闲聊消息）
  ↓
一级子程序执行判断（基于规则 + 轻量分析）：
  ┌─────────────────────────────────────────────────────────────┐
  │                     判断标准矩阵                            │
  ├─────────────────────────────────────────────────────────────┤
  │  ✅ 任务触发（任一项命中即为任务）：                         │
  │                                                             │
  │  1. 包含动作动词：查、找、分析、整理、写、做、改、发、翻译、│
  │     转换、下载、上传、删、统计、计算、监控、设置、配置、生成 │
  │  2. 包含疑问词 + 需要实际操作：如何、怎么（vs 纯知识问答）  │
  │  3. 长度 > 20字 + 包含具体对象（文件/链接/数据等）          │
  │  4. 包含文件名、URL、路径、代码片段                         │
  │  5. 包含时间约束："今天的"、"上周"、"明天" + 具体动作    │
  │  6. 包含操作对象："这个PDF"、"那个Excel"、"我的文档"    │
  │  7. 语气为祈使句开头："帮我把"、"帮我查"、"请把"、"我要"│
  │                                                             │
  │  ❌ 聊天触发（全部命中才判为聊天）：                         │
  │                                                             │
  │  1. 纯问候/告别/天气："早"、"晚安"、"今天好冷"           │
  │  2. 情绪表达/反馈："太棒了"、"无语"、"怎么会这样"        │
  │  3. 纯确认："好的"、"可以"、"嗯"（无附加要求）           │
  │  4. 纯知识问答（无需操作文件或系统）："地球多大"、"Python是"│
  │  5. 闲聊/八卦/日常对话                                     │
  └─────────────────────────────────────────────────────────────┘
  ↓
判断结果：
  ├── **判为聊天** → 一级返回 { "type": "chat", "message": 原消息 }
  │                 主程序接收后走本地 MFDoom 回复
  │                 （省一次 API 调用，免费）
  │
  └── **判为任务** → 进入任务匹配流程
                     ↓
                     匹配任务类型
                     ↓
                     sessions_spawn 对应的二级子程序
```

**边界处理规则：**

| 场景 | 行为 |
|:-----|:-----|
| 模棱两可（既像聊天又像任务） | **宁误不漏** → 判为任务，交给二级去处理 |
| 短句但有操作意图（"翻译"、"查股价"） | 判为任务 |
| 长句但纯情感抒发（"今天真的好烦啊"） | 判为聊天 |
| 先聊天后带任务（"早啊，帮我把那个文件整理一下"） | 整体判为任务 |

### 2.3 二级子程序 (Decomposer) — DeepSeek V4 Flash

| 维度 | 说明 |
|:-----|:-----|
| **模型** | `deepseek/deepseek-v4-flash`（云端按量）|
| **窗口** | 128K |
| **创建方式** | 一级 `sessions_spawn(context="isolated")`，按任务类型路由 |
| **做** | ① 分析任务 ② 拆解为可并行子任务 ③ 创建三级 ④ 合并结果 |
| **不做** | 不做具体执行工作 |

**任务类型与二级子程序的对应关系：**

```
一级匹配到任务类型 → 创建对应的二级子程序
         │
         ├── 📄 文档处理 → 二级(DocumentProcessor)
         │    例：PDF转文字、总结文档、翻译、格式转换
         │
         ├── 🔍 搜索查询 → 二级(SearchAgent)
         │    例：联网搜索答案、查最新资讯、查资料
         │
         ├── 💻 代码编写 → 二级(CodeGenerator)
         │    例：写脚本、写函数、调试代码、代码审查
         │
         ├── ⚙️ 系统操作 → 二级(SystemOperator)
         │    例：查文件、改配置、执行命令、管理进程
         │
         ├── 📊 知识整理 → 二级(KnowledgeOrganizer)
         │    例：整理笔记、归档文档、写MEMORY.md、知识沉淀
         │
         ├── 🖼️ 文件处理 → 二级(FileOperator)
         │    例：处理图片、转换格式、批量重命名
         │
         └── 🧠 复杂推理 → 二级(Reasoner)
              例：数据分析、方案对比、决策建议、逻辑推理
               → 此类任务不拆三级，二级直接完成
```

**注意：** 复杂推理类任务（如方案对比、数据分析）本身不拆成并行子任务，二级直接完成，不创建三级。

### 2.4 三级子程序 (Executor) — DeepSeek V4 Flash

| 维度 | 说明 |
|:-----|:-----|
| **模型** | `deepseek/deepseek-v4-flash`（云端按量）|
| **窗口** | 128K |
| **创建方式** | 二级 `sessions_spawn(context="isolated")` |
| **做** | 具体执行工作（分析、搜索、处理、代码等） |
| **不做** | 不做拆解、不管理状态、不创建后代 |

**三级子程序的特点：**

- 每个三级只做**一件事**（单一职责）
- 三级之间**完全独立**，可并行执行
- 三级返回结果给二级后即完成，不自作主张
- 最多 **5 个三级并行**（受 GTX 960 4GB 限制）
- 三级不需要自己写 `memory`，结果由上级汇总后沉淀

```javascript
    weight: { keyword: 0.5, action: 0.5 },

  "知识整理": {

```

### 3.2 匹配优先级

如果一条消息匹配多个任务类型（如"查PDF总结"同时匹配文档处理和搜索查询），按以下优先级：

```
优先级从高到低：
  1. 系统操作（最危险，优先识别）
  2. 代码编写
  3. 文档处理
  4. 文件处理
  5. 搜索查询
  6. 复杂推理
  7. 知识整理
```

### 3.3 匹配失败处理

```
匹配不到任何任务类型
  ↓
一级子程序：
  ├── 返回 { "type": "chat", "message": 原消息 }
  │   → 主程序走本地 MFDoom 回复
  │
  └── 同时记录日志："未匹配到任务类型的消息：{{message}}"
       → 用于后期优化匹配规则
```

## 四、拆分 + 执行流程（完整示例）

### 示例 1：PDF 转文字 + 总结（文档处理）

```
老板："帮我把这10页PDF转文字总结重点"

步骤 ① — 主程序：
  ├── 判断：不是纯聊天
  ├── sessions_spawn(
  │     model="deepseek/deepseek-v4-flash",
  │     task="判断聊天还是任务+匹配路由。消息: 帮我把这10页PDF转文字总结重点"
  │   )
  └── yield 等待

步骤 ② — 一级子程序：
  ├── 收到消息
  ├── 判断：包含"PDF"+动作动词"转""总结" → 判为任务
  ├── 匹配关键词："PDF" → 文档处理(score:0.8), "总结" → 文档处理(score:0.6)
  ├── 确定任务类型："文档处理"
  ├── 返回主程序：{ "type": "task", "taskType": "文档处理", "message": ... }
  └── 结束

步骤 ③ — 主程序：
  ├── 收到 { "type": "task", "taskType": "文档处理", ... }
  ├── sessions_spawn(
  │     model="deepseek/deepseek-v4-flash",
  │     context="isolated",
  │     task="你是文档处理二级子程序。任务：把10页PDF转文字总结重点。
  │            请将任务拆解为可并行执行的子任务，创建三级子程序执行。
  │            工作步骤：
  │            1. 先分析任务的并行度（每3~4页一个三级）
  │            2. 创建对应数量的三级子程序
  │            3. 每个三级处理一部分页面
  │            4. 接收三级结果 → 合并为完整总结"
  │   )
  └── yield 等待

步骤 ④ — 二级子程序（文档处理）：
  ├── 分析任务：10页PDF，可拆分
  ├── 拆解方案：
  │   ├── 三级A：处理1-3页，提取文字 + 总结要点
  │   ├── 三级B：处理4-6页，提取文字 + 总结要点
  │   ├── 三级C：处理7-10页，提取文字 + 总结要点
  │   └── 最终合并：三级A结果 + B结果 + C结果 → 一份完整总结
  │
  ├── 并行创建三级子程序：
  │   ├── sessions_spawn(
  │   │     model="deepseek/deepseek-v4-flash",
  │   │     task="三级A：处理PDF 1-3页，提取文字并总结要点。返回格式：{\"pages\": \"1-3\", \"text\": ..., \"summary\": ...}"
  │   │   )
  │   ├── sessions_spawn(
  │   │     task="三级B：处理PDF 4-6页..."
  │   │   )
  │   └── sessions_spawn(
  │         task="三级C：处理PDF 7-10页..."
  │       )
  │
  ├── yield 等待所有三级完成（sessions_yield）
  │
  ├── 收到三级结果：
  │   ├── 三级A：{ "pages": "1-3", "text": "...", "summary": "引言+背景介绍" }
  │   ├── 三级B：{ "pages": "4-6", "text": "...", "summary": "核心方法" }
  │   └── 三级C：{ "pages": "7-10", "text": "...", "summary": "实验结果+结论" }
  │
  ├── 合并为完整报告：
  │   {
  │     "summary": "文档共10页，分为三部分：...",
  │     "sections": [
  │       { "title": "引言与背景", "content": "...", "sourcePages": "1-3" },
  │       { "title": "核心方法", "content": "...", "sourcePages": "4-6" },
  │       { "title": "实验与结论", "content": "...", "sourcePages": "7-10" }
  │     ],
  │     "keyTakeaways": ["要点1", "要点2", "要点3"]
  │   }
  │
  ├── 返回结果给一级（结束）
  └── 结束

步骤 ⑤ — 一级获取结果（汇总）：
  ├── 二级已完成，收到合并报告
  ├── 检查结果完整性
  ├── 返回主程序
  └── 结束

步骤 ⑥ — 主程序回复老板：
  → "已处理完10页PDF，以下是总结：\n\n📄 引言与背景(1-3页)：...\n📄 核心方法(4-6页)：...\n📄 实验与结论(7-10页)：...\n\n💡 关键要点：\n• 要点1\n• 要点2\n• 要点3"
```

### 示例 3：联网搜索 + 整理（搜索查询 + 知识整理组合）

```
步骤 ② — 二级（搜索查询）：
  ├── 拆解子任务：
  │   ├── 三级A：搜索 2026年AI芯片 NVIDIA/AMD/Intel 新品

  │   └── 三级D：搜索 2026年AI芯片 市场格局/出货量
  │
  ├── 并行创建三级A-D
  ├── yield 等待
  ├── 合并搜索结果
  └── 返回一级：{ "type": "search_results", "data": { ... } }

步骤 ③ — 一级收到结果：
  ├── 发现需要整理成报告
  ├── 创建"知识整理"二级：
  │   sessions_spawn(
  │     task="将以下搜索资料整理为结构化报告：{...}"
  │   )
  └── yield 等待

步骤 ④ — 二级（知识整理）：
  ├── 整理为完整报告
  └── 返回给一级

步骤 ⑤ — 主程序回复老板

> 注意：示例 3 展示了"链式二级"场景——一级在搜索完成后，再调用知识整理二级进一步处理。这是为了跨二级任务类型的协作，由一级主动调度。
```

### 3.4 不可拆分任务的处理

有些任务不适合拆成并行子任务，二级直接处理：

| 任务类型 | 是否可拆分 | 原因 |
|:---------|:----------:|:-----|
| 写一个脚本/函数 | ❌ 不拆分 | 代码前后依赖，拆了没法合并 |
| 对比分析两个方案 | ❌ 不拆分 | 需要全局视角判断 |
| 查当前时间/天气 | ❌ 不拆分 | 一步操作，拆了浪费 |
| PDF转文字(长文档) | ✅ 可拆分 | 按页并行处理再合并 |
| 搜索多个关键词 | ✅ 可拆分 | 各关键词独立搜索 |
| 批量处理多个文件 | ✅ 可拆分 | 每个文件一个三级 |
| 系统任务（改配置等） | ❌ 不拆分 | 原子操作，拆了有风险 |

## 五、通信协议

### 5.1 层级间返回格式

所有子程序必须使用统一的 JSON 格式返回结果：

```jsonc
// 一级 → 主程序 的返回格式
{
  "type": "chat" | "task",   // chat=纯聊天, task=任务指令
  "confidence": 0.0-1.0,     // 判断置信度
  "message": "原始消息",       // 原消息原文
  "taskType": "文档处理",     // 仅 type=task 时有
  "matchedKeywords": ["PDF", "总结"],  // 命中的关键词
  "result": { ... }           // 最终执行结果（仅最终汇总时）
}

// 二级 → 一级 的返回格式
{
  "taskType": "文档处理",
  "subtasks": 3,             // 拆分子任务数
  "result": {                 // 合并后的完整结果
    "summary": "...",
    "details": [...],
    "durationMs": 45000
  },
  "errors": [],               // 错误列表（如果有三级失败）
  "failedCount": 0            // 三级失败数量
}

// 三级 → 二级 的返回格式
{
  "id": "三级A",
  "task": "处理PDF 1-3页",
  "completed": true,
  "result": {                 // 具体执行结果
    "text": "...",
    "summary": "...",
    "pages": "1-3"
  },
  "durationMs": 15000,
  "error": null               // 或 { "message": "...", "code": "..." }
}
```

### 5.2 完整通信流程

```
[老板]                       [主程序]                [一级]              [二级]                  [三级]
  │                            │                      │                 │                      │
  │── 消息 ───────────────────→│                      │                 │                      │
  │                            │── sessions_spawn ───→│                 │                      │
  │                            │                      │── 判断+匹配      │                      │
  │                            │                      │── sessions_spawn ─→│                     │
  │                            │                      │                  │── 分析+拆解           │
  │                            │                      │                  │── sessions_spawn ────→│ (三级A)
  │                            │                      │                  │── sessions_spawn ────→│ (三级B)
  │                            │                      │                  │── sessions_spawn ────→│ (三级C)
  │                            │                      │                  │                      │
  │                            │                      │                  │◄── yield 等待完成 ────│
  │                            │                      │                  │◄── yield 等待完成 ────│
  │                            │                      │                  │◄── yield 等待完成 ────│
  │                            │                      │                  │                      │
  │                            │                      │◄── 合并结果 ─────│                      │
  │                            │◄── 汇总结果 ────────│                  │                      │
  │◄── 回复 ──────────────────│                      │                  │                      │
```

### 5.3 错误处理

| 故障位置 | 处理方式 |
|:---------|:---------|
| **三级失败** (1个三级挂掉) | 二级继续等待其他三级完成，失败的标记为 error。合并时排除失败部分，备注"第X部分处理失败" |
| **三级失败** (超过一半三级挂掉) | 二级停止等待 → 返回 partial result + 错误说明。一级收到后标注"部分失败" |
| **二级失败** | 一级收到错误 → 重试二级（最多2次）。仍失败 → 返回主程序，主程序回复老板"处理失败：原因" |
| **一级失败** | 主程序收到一级 err 状态 → 重试（最多2次）。仍失败 → 主程序回复老板"系统处理异常，请重试" |
| **所有三级超时** | 二级有超时监控（默认 300 秒）。超时未完成的标记为 failed + reason:timeout |
| **三级返回格式不对** | 二级先做格式校验，格式异常的标记错误，不阻塞合并流程 |

**超时时间配置：**

| 层级 | 默认超时 | 说明 |
|:-----|:--------:|:-----|
| 一级 | 30 秒 | 判断 + 匹配 + 路由，不应太久 |
| 二级 | 60 秒 | 拆解 + 创建三级，如果三级多则等三级完成 |
| 三级 | 300 秒 | 具体执行，取决于任务复杂度 |

## 七、与现有架构的关系

### 7.1 与 Route Server 的协同

三级子程序架构与 Route Server 架构是 **互补关系**，不是替代关系：

```
              Route Server（宏观状态管理）
              ┌────────────────────────┐
              │ • 话题聚合（去重）       │
              │ • 任务队列管理           │
              │ • 状态持久化（崩溃恢复）  │
              │ • GUI 实时监控           │
              │ • 决策管理器（老板选方案） │
              └──────────┬─────────────┘
                         │ 提供：任务ID、状态、队列
                         │ 读取：子程序状态、进度
                         ▼
              三级子程序架构（微观执行流程）
              ┌────────────────────────┐
              │ • 聊天vs任务判断         │  ← 一级
              │ • 任务类型匹配路由       │  ← 一级
              │ • 任务拆解               │  ← 二级
              │ • 并行子任务执行         │  ← 三级
              │ • 结果合并               │  ← 二级
              └────────────────────────┘
```

**协同方式：**

| 场景 | Route Server 负责 | 三级架构负责 |
|:-----|:-----------------|:------------|
| 新消息到达 | 查话题注册表，决定新建合并 | 一级判断聊天vs任务 |
| 任务运行中 | 持久化状态到 JSON，GUI 显示 | 二级拆解 + 三级执行 |
| 需要老板决策 | 管理决策文件 + 推送 + 回传 | 三层不需要处理决策 |
| 任务完成 | 更新状态到完成队列 | 结果逐级上传 |
| 系统崩溃 | 恢复任务队列 | 任务重新调度 |

### 7.2 决策推送在哪一层触发

决策推送在 **二级或三级** 遇到需要老板选择时触发：

```
场景：三级在执行时发现需要老板决定方案

三级A（处理PDF时发现无法识别字体）
  → 返回二级：{ "id": "三级A", "completed": false, "needsDecision": true,
                "decision": { "prompt": "PDF字体无法识别，选方案：",
                              "options": [
                                { "key": "A", "label": "忽略字体直接提取文字" },
                                { "key": "B", "label": "用 OCR 识别" }
                              ]} }
  → 二级收到需要决策
     → 二级调用 Route Server 决策接口
       POST /api/decisions { taskId, prompt, options }
     → 等待老板回复（通过 GUI 或聊天通道）
     → 收到回复后，继续执行
     → 如果此时其他三级已完成，先合并已完成的部分
```

**决策触发流程归纳：**

```
三级需要决策 → 返回 needsDecision=true 给二级
  → 二级调用 Route Server 决策管理器
    → GUI 弹出 + 聊天推送
    → 老板回复 → Route Server 回传
    → 二级继续（或通知三级继续）
```

### 7.3 费用记录位置

API 调用费用记录由**一级子程序**负责写入「柒月的知识库」→「AI费用账单」：

```
一级在汇总完成后，一次性记录本次任务的全部费用：
  - 一级自身调用（判断+匹配）
  - 二级调用
  - 所有三级的调用
```

```
  ⑤ 二级合并 → 一级汇总 → 主程序回复
验证：3个三级同时运行，结果合并正确
```

### 测试用例 4：单三级失败容错

```
输入："帮我把这些文件整理一下"（3个文件，其中一个损坏）
预期：
  ① 拆成3个三级（每个处理1个文件）
  ② 三级B（损坏文件）失败 → 返回 error
  ③ 二级收到：三级A+C成功，三级B失败
  ④ 二级合并：包含三级A+C结果 + 说明"文件B处理失败，原因：文件损坏"
  ⑤ 最终回复包含成功结果和失败说明
验证：部分失败不阻塞整体流程
```

## 十、关键规则与注意事项

### ✅ 必须遵守

1. **主程序不做具体工作**：只做简单聊天判断 + 转发 + 回复，不分析不计算
2. **一级必须是 "chat" vs "task" 的守门人**：模棱两可一律判任务
3. **二级拆解时最小粒度**：每个三级只做一件事，超过就要再拆
4. **三级最多 5 个并行**：GTX 960 4GB 限制，超了排队
5. **失败容错**：部分三级失败不影响整体，但必须标注
6. **统一返回格式**：所有层级用 JSON 通信

### ❌ 不能做

1. **三级不能 spawn 后代**：三级执行完就结束，不创建子子程序
2. **一级不能绕过二级直接创建三级**：除非任务简单到一级直接处理（但这种情况实际上应该由主程序本地回复）
3. **主程序不能跳过一级直接创建二级**：必须经过一级路由
4. **不能把聊天当任务处理**：聊天的代价是 DeepSeek API 费用，浪费钱
