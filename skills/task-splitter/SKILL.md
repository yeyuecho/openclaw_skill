# Task Splitter — 智能任务拆分引擎

## 概述

当收到一个复杂任务时，本技能将其自动分解为多个独立的子任务，每个子任务通过 `sessions_spawn` 创建独立的子会话并行执行。由于每个子任务的上下文都很短（3-5 条消息），**全程可用本地轻量模型**，不溢出，不费钱。

---

## 核心流程

```
收到复杂任务
     │
     ▼
┌─────────────────┐
│   ① 任务分析     │ ← 判断是否能拆分成独立子任务
└────────┬────────┘
         │ 可拆分
         ▼
┌─────────────────┐
│   ② 任务分解     │ ← 拆成 N 个独立子任务，每个一个目标
└────────┬────────┘
         │
         ▼
┌──────────────────────────────────────┐
│   ③ 并行执行                          │
│  ┌────────┐  ┌────────┐  ┌────────┐  │
│  │ 子任务1 │  │ 子任务2 │  │ 子任务N │  │  ← 各自 sessions_spawn
│  │ 本地模型 │  │ 本地模型 │  │ 本地模型 │  │     3-5条消息，不溢出
│  └────────┘  └────────┘  └────────┘  │
└────────┬──────────────────────────────┘
         │ sessions_yield 等待全部完成
         ▼
┌─────────────────┐
│   ④ 汇总输出     │ ← 合并所有子结果，整理给用户
└─────────────────┘
```

## 子智能体模型分配规则

| 场景 | 模型 | 原因 |
|------|------|------|
| 大多数子任务（文件分析、代码编写、逻辑推理等） | `ollama/deepseek-r1:1.5b`（本地） | 子任务上下文短，4K 窗口够用，免费 |
| 需要联网搜索 | `deepseek/deepseek-v4-flash`（云端） | 需要工具调用能力强 |
| 需要 API 调用 / 网络请求 | `deepseek/deepseek-v4-flash`（云端） | 需要工具调用能力强 |
| 用户明确要求用云端 | `deepseek/deepseek-v4-flash`（云端） | 用户指令优先 |
| 子任务在本地模型 fail 过至少 1 次 | `deepseek/deepseek-v4-flash`（云端） | 避免反复失败 |

## 任务拆分原则

### 能拆分的任务特征
- ✅ 多个互不依赖的文件需要同时处理
- ✅ 批量数据处理（下载、转换、提取）
- ✅ 多步骤流程（步骤间依赖低）
- ✅ 同一问题涉及不同维度（技术+成本+时间）

### 不能/不建议拆分的任务特征
- ❌ 单步骤依赖链（先 A 才能 B）
- ❌ 共享上下文很重要（子任务间需要知道对方结果）
- ❌ 简单一句话就能回答的

## 使用示例

### 用户："分析这 10 个文件，提取关键数据"

```
拆分：
  子任务 1-10：每个文件单独分析 → 各走本地模型（3条消息，轻松）
  主会话：合并 10 份结果 → 走本地模型（仅做汇总，不超限制）
```

### 用户："帮我在网上查最新的显卡价格，然后比较优劣"

```
拆分：
  子任务 1：搜 NVIDIA 40/50 系列价格 → 云端（需要联网）
  子任务 2：搜 AMD 7000/9000 系列价格 → 云端（需要联网）
  子任务 3：对比两者优劣 → 本地模型（纯分析，无需联网）
  主会话：合并对比结果 → 本地模型（仅编排文本）
```

---

## 注意事项

- 主会话模型也走本地（`ollama/deepseek-r1:1.5b`），仅做调度和汇总
- 子智能体间**不共享上下文**，各自独立
- 结果通过 `completion event` 回传，不需轮询
- 使用 `sessions_yield` 等待全部完成，不要 `subagents list` 轮询
- 当主会话消息数 ≥ 30 条时，主动 spawn 新会话来处理下一个复杂任务，避免上下文堆积
